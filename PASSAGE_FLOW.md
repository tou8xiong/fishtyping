# FishTyping - Passage Generation & Flow

## Overview

This document explains how the typing passage generation and data flow works in FishTyping.

## Architecture

### 1. Passage Generation Flow

```
User Opens App
    ↓
TypingTest Component Loads
    ↓
Calls /api/generate-passage (POST)
    ↓
API checks database for available passage
    ↓
┌─────────────────────────────────────┐
│ Passage found in DB?                │
├─────────────────────────────────────┤
│ YES → Return from DB                │
│ NO  → Generate with Gemini AI       │
│       Save to DB                    │
│       Return to user                │
└─────────────────────────────────────┘
    ↓
User types the passage
    ↓
On completion → Save to passage_history
    ↓
Update user stats
```

### 2. Database Tables

#### `passages`
Stores pre-generated and AI-generated typing passages.

```sql
- id (uuid)
- content (text)
- language (enum: english, lao)
- difficulty (enum: beginner, intermediate, advanced, expert)
- length (enum: short, medium, long)
- theme (enum: general, technology, nature, science, history)
- challenge_type (enum: standard, punctuation, numbers, speed)
- status (enum: generating, ready, in_use, archived)
- generated_by (enum: manual, ai)
- ai_model (text)
- used_count (integer)
- word_count (integer)
- created_at (timestamp)
- updated_at (timestamp)
```

#### `passage_history`
Tracks user performance on completed passages.

```sql
- id (uuid)
- user_id (uuid) → references auth.users
- passage_id (uuid) → references passages
- attempted_at (timestamp)
- wpm (integer)
- accuracy (integer)
- duration_ms (integer)
```

#### `users`
User profile information.

```sql
- id (uuid) → references auth.users
- username (text)
- display_name (text)
- avatar_url (text)
- preferred_language (text)
- created_at (timestamp)
- updated_at (timestamp)
```

#### `generation_jobs`
Queue for background passage generation.

```sql
- id (uuid)
- status (enum: pending, processing, completed, failed)
- priority (integer)
- language (enum)
- difficulty (enum)
- length (enum)
- theme (enum)
- challenge_type (enum)
- attempts (integer)
- error_message (text)
- created_at (timestamp)
- processed_at (timestamp)
```

### 3. API Routes

#### `POST /api/generate-passage`
Generates or retrieves a typing passage.

**Request:**
```json
{
  "difficulty": "intermediate",
  "length": "medium",
  "theme": "general",
  "challengeType": "standard",
  "language": "english"
}
```

**Response:**
```json
{
  "passage": "The text content...",
  "passageId": "uuid",
  "source": "db" | "ai"
}
```

#### `POST /api/worker/generate-pool`
Background worker to pre-generate passages.

**Response:**
```json
{
  "success": true,
  "generated": 10,
  "message": "Generated 10 passages"
}
```

#### `GET /api/worker/generate-pool`
Check passage pool status.

**Response:**
```json
{
  "poolStatus": [
    {
      "difficulty": "intermediate",
      "length": "medium",
      "theme": "general",
      "challengeType": "standard",
      "count": 5
    }
  ],
  "minPoolSize": 5
}
```

### 4. Pre-Generation System

The system maintains a pool of pre-generated passages to ensure fast response times.

**Minimum Pool Size:** 5 passages per combination

**Total Combinations:**
- 4 difficulties × 3 lengths × 5 themes × 4 challenge types × 2 languages = 480 combinations
- Minimum passages needed: 480 × 5 = 2,400 passages

**Background Worker:**

Run manually:
```bash
node scripts/generate-passages.js
```

Or set up a cron job:
```bash
# Run every hour
0 * * * * cd /path/to/fishtyping && node scripts/generate-passages.js >> /var/log/fishtyping-worker.log 2>&1
```

Or use a service like Vercel Cron Jobs:
```json
// vercel.json
{
  "crons": [{
    "path": "/api/worker/generate-pool",
    "schedule": "0 * * * *"
  }]
}
```

### 5. User Flow

#### Unlogged User
1. Visit homepage → See typing test
2. Can practice typing (results not saved)
3. Click "Login" → Go to /login
4. After login → Redirect to homepage
5. Results now saved to database

#### Logged User
1. Visit homepage → See typing test
2. Select difficulty, language, etc.
3. Click "Start Typing"
4. Type the passage
5. On completion:
   - Stats calculated (WPM, accuracy, time)
   - Saved to `passage_history`
   - Can view in profile/leaderboard

### 6. Key Functions

#### `getAvailablePassage()`
Retrieves a passage from the database that:
- Matches the requested parameters
- Has status 'ready'
- User hasn't completed before (if logged in)
- Has lowest usage count

#### `saveGeneratedPassage()`
Saves a newly generated passage to the database with:
- Content and metadata
- Status: 'ready'
- Generated by: 'ai'
- AI model: 'gemini-2.0-flash'

#### `trackPassageResult()`
Saves user performance to `passage_history`:
- User ID
- Passage ID
- WPM, accuracy, duration

### 7. Client-Side Components

#### `TypingTest.tsx`
Main typing test component:
- Loads passages from API
- Tracks typing progress
- Saves results on completion

#### `useTypingEngine.ts`
Custom hook for typing logic:
- Calculates WPM, accuracy
- Tracks time elapsed
- Detects completion

#### `passageGenerator.ts`
Utility functions:
- Client-side Gemini API calls (fallback)
- Passage tracking functions
- Result saving

## Troubleshooting

### Passages not loading
1. Check database connection
2. Verify Gemini API key
3. Check browser console for errors
4. Verify `/api/generate-passage` endpoint

### Results not saving
1. Verify user is logged in
2. Check `passage_history` table permissions
3. Check browser console for errors

### Worker not generating passages
1. Verify `GEMINI_API_KEY` is set
2. Check API rate limits
3. Review worker logs
4. Verify database write permissions
